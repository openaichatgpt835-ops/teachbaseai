# KB Eval Pack

Цель: держать стабильное качество ответов RAG и не ломать поведение при следующих правках.

## Базовые сценарии

1. Арифметика без базы знаний
- Запрос: `сколько будет три плюс два`
- Ожидание:
  - быстрый локальный ответ
  - без обращения к базе знаний
  - ответ вида `Ответ: 5`

2. Numeric/factual запрос без релевантного контекста
- Запрос: `сколько жмет спартак`
- Ожидание:
  - нельзя брать случайные числа из нерелевантных документов
  - нельзя отвечать promo/скидками/датами из мусорных чанков
  - либо точный grounded ответ, либо честный дефицит данных

3. Product-grounded instruction
- Запрос: `как настроить доступ к базе знаний`
- Ожидание:
  - ответ опирается на найденные шаги/разделы продукта
  - без общих best practices про роли, папки и процессы
  - если шаги есть, формат `1) 2) 3)`

4. Overview / broad question
- Запрос: `расскажи мне про тави и её спутников`
- Ожидание:
  - обзорный ответ строится по связанным сущностям из контекста
  - не выдумываются новые имена/персонажи
  - ответ шире одной фразы, если в контексте достаточно материала

5. List / top-N
- Запрос: `дай топ 5 самых мощных заклинаний и инструкции как их сотворить`
- Ожидание:
  - список остается списком
  - нет агрессивного схлопывания в 1-2 пункта
  - нет сырых dump-фраз из чанков
  - нет чрезмерной воды и перестраховочных абзацев

## Что считаем регрессией

- потеря ответа при серии быстрых сообщений в web chat
- numeric ответ из нерелевантного документа
- overview с галлюцинациями новых сущностей
- instruction ответ с общими внешними рекомендациями
- list-ответ, который:
  - теряет форму списка
  - становится сухим поисковым перечнем
  - или снова начинает подменяться сырыми фразами из чанков
